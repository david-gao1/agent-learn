# Chapter 1 Deep Dive Questions

## 第 1 章讲了什么

第 1 章的核心不是介绍某个具体框架，而是建立一个视角：LLM 应用的重心正在从 Prompt Engineering 转向 Context Engineering，再进一步走向 Harness Engineering。

它分三层展开：

1. **上下文工程的本质**：不是把提示词写漂亮，而是把用户意图、背景知识、工具反馈、历史记忆组织成模型可用的工作语境。
2. **上下文工程的结构**：成熟系统围绕感知、规划、执行、反思四个模块构建上下文，并通过记忆管理与上下文卸载控制噪声。
3. **Agent 的演进路线**：从 Function Calling、ReAct，到深度思考型 Agent、Code Agent、多 Agent，背后都在解决一个问题：如何让模型在更长、更复杂的任务中持续保持有效上下文。

一句话概括：第 1 章是在回答“为什么 Agent 工程的重点，不再是一次提示，而是持续构建、管理和隔离上下文”。

## 可以深挖的 4 个问题

### 问题 1：Prompt Engineering 和 Context Engineering 的真正分界在哪里？

**为什么值得深挖**

很多人会把上下文工程误解成“写更长的提示词”。第 1 章通过购车案例说明，真正的变化不是句子变长，而是信息来源、信息结构和决策过程发生了变化。

**可以探索到哪一层**

- 表层：Prompt 是表达技巧，Context 是信息组织。
- 中层：Context 包含意图、边界条件、外部知识、工具反馈、记忆。
- 深层：Context Engineering 的目标不是让模型“回答得更像”，而是让模型在任务过程中保持目标一致、信息充分、噪声可控。

**适合写成的文章**

《如何判断一个系统是否真的进入上下文工程》

**文章边界**

只讨论 Prompt 与 Context 的分界，不展开 Agent Loop、Skills、Memory 的完整系统。最多用它们作为后文预告。

**优化提醒**

这篇文章最容易散成“提示词技巧大全”。应该只围绕一个判断推进：如果信息仍然由人一次性塞给模型，那仍偏 Prompt；如果系统能持续收集、筛选、压缩、更新信息，才进入 Context Engineering。

### 问题 2：为什么上下文工程天然需要 Agent？

**为什么值得深挖**

第 1 章把 ReAct 解释为动态构建上下文的过程。这个视角很重要：Agent 不是在上下文工程之外的东西，而是上下文工程的执行载体。

**可以探索到哪一层**

- 表层：Agent 能调用工具，所以能补充信息。
- 中层：每次工具观察都会成为下一轮推理的上下文。
- 深层：Agent 的价值在于让上下文从静态输入变成动态演化过程。

**适合写成的文章**

《Agent 不是会调用工具的聊天机器人，而是上下文构建机器》

**文章边界**

只讲 Agent 与 Context 的关系，不深入比较所有 Agent 框架。可以用 ReAct 作为主线：Reason -> Act -> Observe -> Context Update。

**优化提醒**

这篇文章要避免变成 Agent 分类文章。重点不是“有哪些 Agent”，而是“为什么 Agent 的每一步行动都在改变模型下一步能看到什么”。

### 问题 3：上下文构建为什么需要感知、规划、执行、反思四个模块？

**为什么值得深挖**

这四个模块是第 2 章工程实践的理论入口。第 1 章先给框架，第 2 章再逐个实现。把这里吃透，后面读意图识别、计划模式、反思模式、CodeAct 会轻很多。

**可以探索到哪一层**

- 感知：把自然语言变成结构化任务意图。
- 规划：决定信息收集与动作顺序。
- 执行：通过工具拿到实时观察。
- 反思：检查信息缺口、错误路径和遗漏。
- 更深一层：这四个模块不是功能清单，而是上下文生命周期的四个阶段。

**适合写成的文章**

《如何用感知、规划、执行、反思控制上下文质量》

**文章边界**

只解释四模块如何共同构建上下文，不展开每个模块的完整代码实现。每个模块只给一个工程例子。

**优化提醒**

这篇文章要防止平均用力。建议抓住一个贯穿案例，比如“让 Agent 分析一个代码仓库”或“让 Agent 写一份研究报告”，用同一个案例穿过四个模块。

### 问题 4：多 Agent 的本质是分工，还是上下文隔离？

**为什么值得深挖**

第 1 章对多 Agent 的解释很有价值：常见说法是“工具太多，所以要分专家”；但从上下文工程看，更深的原因是单 Agent 上下文会污染、膨胀、耗尽。

**可以探索到哪一层**

- 表层：SubAgent 负责专项任务。
- 中层：Manager Agent 只接收结果，不承接全部过程噪声。
- 深层：多 Agent 是一种上下文窗口隔离技术，让不同任务拥有干净的语义边界。

**适合写成的文章**

《SubAgent 的真正价值：不是更多人手，而是更干净的上下文》

**文章边界**

只讨论 SubAgent/Multi-Agent 与上下文污染，不扩展到组织管理、团队协作、复杂 MAS 理论。

**优化提醒**

这篇文章最容易讲成“多 Agent 很强”。要反过来写：先描述单 Agent 为什么会被中间结果、工具说明、历史轨迹淹没，再说明 SubAgent 如何把噪声隔离掉。

## 第 1 章之后，最推荐先写哪一篇

优先写问题 1：

《如何判断一个系统是否真的进入上下文工程》

原因：

- 它是所有后续文章的地基。
- 它能解释为什么这本书从 Prompt 讲起，却很快进入 Agent、Memory、Skills、SubAgent。
- 它的边界最清楚，不容易过早展开到完整 Harness 系统。

建议文章只回答一个问题：**上下文工程到底比提示词工程多了什么？**

## 每篇文章的注意力优化原则

写完每篇文章后，固定问 5 个问题：

1. 这篇文章的中心问题能否用一句话说清？
2. 每一节是否都在推进这个中心问题？
3. 有没有只是因为“也相关”而放进来的内容？
4. 有没有一个贯穿案例帮助读者减少切换成本？
5. 读者看完后，只需要记住的那个结论是什么？

如果一篇文章同时想解释 Prompt、Agent、Memory、Skills、SubAgent，它大概率会散。更好的做法是：每篇只深挖一个机制，其他概念只在必要时作为背景出现。
