# MiniClaw 练习路线

这份路线不是阅读清单，而是用来训练判断力。

每个练习都按同一个顺序做：

1. 先回答“我预期系统会怎么做”。
2. 再运行命令拿证据。
3. 最后回到代码，确认机制在哪里实现。

如果只跑命令但不预测，你看到的是功能。如果先预测再验证，你练到的是 Harness Agent 的工程直觉。

## 准备

先生成一份完整证据包：

```bash
make walkthrough
```

如果只想快速验收学习闭环：

```bash
make learn-check
```

输出目录通常是：

```text
walkthrough-output/miniclaw-<timestamp>/
```

下面的练习都基于这个目录里的文件。也可以照着 `docs/harness-agent-learning/miniclaw-walkthrough.md` 手动运行命令。

## 练习 1：判断 Agent Loop 是否真的发生

先回答：

- 如果 Agent 真的在行动，trace 里至少应该出现哪些事件？
- 哪个事件证明它不是“一次性生成答案”？
- 哪个事件证明它观察过环境？

再查看：

```bash
cat walkthrough-output/miniclaw-*/仓库分析-trace.txt
```

如果是手动运行 CLI，可以只看观察事件：

```bash
python3 prototypes/miniclaw_harness/main.py --db "$DB" trace-show "$TASK_ID" --event observation
```

也可以只看行动和观察这条线：

```bash
python3 prototypes/miniclaw_harness/main.py --db "$DB" trace-show "$TASK_ID" --event tool_call --event observation
```

你要找的证据：

- `plan`
- `decision`
- `tool_call`
- `observation`
- `final_result`

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/runtime.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/store.py`

掌握标准：

你能说清楚为什么 `tool_call -> observation` 是 Agent 从“回答”进入“行动”的边界。

## 练习 2：判断 Tool 边界在哪里限制风险

先回答：

- FileTool 应该阻止什么风险？
- BashTool 为什么不能直接开放 shell？
- CodeTool 为什么要做 AST 检查？

再查看测试：

```bash
python3 -m unittest prototypes/miniclaw_harness/tests/test_miniclaw_harness.py -v
```

重点搜索这些测试名：

```bash
rg "blocks|allowlisted|restricted|unsafe|escape" prototypes/miniclaw_harness/tests/test_miniclaw_harness.py
```

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/tools.py`

掌握标准：

你能把“模型想做什么”和“Harness 允许做什么”分开讲清楚。

## 练习 3：区分 Trace、State、Memory

先回答：

- Trace 适合保存什么？
- State 适合保存什么？
- Memory 为什么不是聊天历史？

再查看：

```bash
cat walkthrough-output/miniclaw-*/仓库分析-trace.txt
cat walkthrough-output/miniclaw-*/仓库分析-state.txt
cat walkthrough-output/miniclaw-*/memory.txt
```

判断：

- 哪些内容是过程证据？
- 哪些内容是恢复任务需要的结构化状态？
- 哪些内容是完成后可复用的经验？

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/store.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/runtime.py`

掌握标准：

你能说明为什么上下文工程不是“把更多文本塞进 prompt”，而是把信息放到正确的载体里。

## 练习 4：解释 Skills 的渐进加载

先回答：

- 为什么不在一开始加载完整 Skill？
- Skill 的标签、说明、正文分别适合在什么时候进入上下文？
- Skill 和普通 prompt 模板的差异在哪里？

再查看：

```bash
cat walkthrough-output/miniclaw-*/仓库分析-state.txt
cat prototypes/minimal_harness_agent/skills/repo-reading/SKILL.md
```

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/skills.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/runtime.py`

掌握标准：

你能解释 Skills 为什么是“上下文按需加载机制”，而不只是“提示词文件夹”。

## 练习 5：把 Human Approval 当成状态看

先回答：

- 人类审批如果只是口头确认，会丢失哪些证据？
- 一个可恢复的审批任务至少要保存什么？

再查看：

```bash
cat walkthrough-output/miniclaw-*/approval-trace.txt
```

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/background.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/store.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/runtime.py`

掌握标准：

你能说明 Human-in-the-loop 在 Harness 里不是“人在旁边看”，而是任务状态机的一部分。

## 练习 6：解释 CodeAct 的工程边界

先回答：

- CodeAct 解决了什么问题？
- 为什么 CodeAct 的核心不是“让模型写代码”，而是“让代码执行受控”？
- 如果模型生成了不安全代码，Harness 应该怎么处理？

再查看：

```bash
cat walkthrough-output/miniclaw-*/codeact-trace.txt
cat walkthrough-output/miniclaw-*/codeact-state.txt
```

回到代码：

- `prototypes/miniclaw_harness/src/miniclaw_harness/tools.py`
- `prototypes/miniclaw_harness/src/miniclaw_harness/runtime.py`

掌握标准：

你能把 CodeAct 的能力和风险同时讲出来：代码让 Agent 更强，工具边界让它可控。

## 练习 7：做一次小改造

目标：

给 MiniClaw 增加一个很小的新证据字段或 trace 事件。

已完成示例：

- 仓库分析 State 增加了 `tools_used`。
- 这个字段属于 State，因为它让恢复和复盘任务时能稳定看到一次任务实际越过了哪些工具边界。
- 取值示例包括 `FileTool.list_files`、`FileTool.read_file`、`BashTool.run`。
- 仓库分析 State 增加了 `last_observation_at`。
- 这个字段属于 State，因为它把最近一次环境观察的 trace 时间沉淀成可恢复线索。
- CodeAct state 增加了 `code_safety_status`。
- 这个字段属于 State，因为它是恢复和复盘 CodeAct 任务时需要稳定读取的结构化判断。
- 取值包括 `trusted_rule`、`accepted`、`rejected_fallback`，分别对应规则代码、模型代码通过 Harness 校验、模型代码被拒绝后使用 fallback。
- 任务报告增加了 `State Field Notes`。
- 这个说明段属于 Report，因为它把结构化字段翻译成复盘时能直接理解的工程含义。
- `memory-list` 增加了来源任务 State 摘要。
- 这个摘要属于 Memory 的检查入口，因为长期记忆需要能回到产生它的任务状态，才能判断这条经验是否可信、是否可复用。
- `background-list` 增加了 `next_step` 提示。
- 这个提示属于 Task System，因为等待审批和阻塞任务都需要把恢复动作暴露给操作者，而不是只显示一个静态状态。
- `trace-show` 增加了 `--event` 过滤参数。
- 这个过滤属于 Trace 检查入口，因为学习 Agent Loop 时经常需要只看 `tool_call -> observation` 这条行动反馈链。
- `task-report` 增加了 `--summary` 简短模式。
- 这个摘要属于 Report，因为它把完整证据压缩成机制、行动边界、状态证据和复盘重点。
- `task-report --summary` 增加了 repo analysis 的端到端摘要证据。
- 这个字段属于 Report，因为它用 `loop_evidence` 把多步任务压缩成 `plan -> tool_call -> observation -> final_result`。
- `task-report --summary` 增加了 Human Approval 摘要证据。
- 这个字段属于 Report，因为它把 `approval_status=pending` 和 `next_step=approve-task` 放到同一个复盘视图里。
- `task-report --summary` 增加了 blocked recovery 摘要证据。
- 这个字段属于 Report，因为它把 `test_status=failed` 和 `next_step=resume-task` 放到同一个复盘视图里。
- MiniClaw 增加了 `learn-check` 短学习验收。
- 这个命令属于学习入口，因为它用一个临时 workspace 快速验证 Agent Loop、CodeAct、Memory、Human Approval 和 Summary 是否都可观察。

后续可选题目：

- 给 `learn-check` 增加 Compact 验收项。

要求：

1. 先写一个失败测试。
2. 再改实现。
3. 跑 `make verify`。
4. 用一段话说明这个字段属于 Trace、State 还是 Memory。

掌握标准：

你不只是能解释 Harness 组件，还能判断一个新信息应该沉淀到哪里。

## 练习完成后的自测

不用背定义，回答这些问题：

- 一个多步 Agent 任务为什么需要 trace？
- 为什么 state 比长聊天历史更适合恢复任务？
- 什么情况下应该用 SubAgent 隔离上下文？
- Tool 的安全边界应该靠 prompt 还是代码？
- Skills 为什么要渐进加载？
- CodeAct 的失败应该如何被记录和恢复？

能回答这些问题，就可以进入下一步：自己扩展 MiniClaw，而不是继续堆概念。
