# 从上下文工程到 Harness Agent：一个工程师视角的系统理解

## 1. 为什么 Prompt 不够

Prompt Engineering 关注的是如何写好一次输入。它能提升单轮回答质量，但很难解决真实工程任务中的几个问题：任务会变长，信息会变多，工具调用会产生新观察，用户意图会变化，模型本身不会自动保存状态。

Context Engineering 关注的是另一件事：每一次模型调用时，应该把哪些信息、以什么结构、按什么优先级放进上下文。它把问题从“怎么问”推进到“怎么组织模型的工作记忆”。

展开文章：[不是提示词变长了，而是任务语境被工程化了](./chapter-1/01-不是提示词变长了，而是任务语境被工程化了.md)。

## 2. Agent 的核心不是会聊天，而是能持续行动

Agent 的关键能力来自循环。一个最小 Agent Loop 包含四步：

1. 读取当前目标和上下文。
2. 决定下一步行动。
3. 调用工具并观察结果。
4. 更新上下文并继续。

这让 Agent 从一次性回答变成一个可以持续工作的系统。

## 3. 五种基础设计模式

- **意图识别**：在入口判断用户想做什么、是否安全、该走哪条流程。
- **计划模式**：先生成计划，再按步骤执行，适合多步骤任务。
- **反思模式**：生成结果后自我检查并重试，适合高可靠性任务。
- **CodeAct**：让模型生成代码并执行，把代码作为通用工具。
- **Human-in-the-loop**：在高风险或不确定节点让人审核、确认或接管。

## 4. Harness Agent 的核心组件

- **Loop**：让 Agent 能连续工作。
- **Tools**：让 Agent 能观察和改变外部环境。
- **Context**：决定模型当前知道什么。
- **Skills**：把专家经验包装成可渐进加载的能力。
- **Memory**：把事实、偏好、总结和任务状态留到未来。
- **SubAgent**：把复杂执行隔离到独立上下文，减少主 Agent 噪声。
- **Task System**：让任务可追踪、可恢复、可审计。

## 5. 最小原型说明

本仓库的 `prototypes/minimal_harness_agent/` 是一个离线原型。它不调用真实 LLM，而是用确定性流程展示 Harness 结构：

- 创建任务。
- 根据请求选择 Skill。
- 使用受限 shell 工具观察工作区。
- 压缩上下文。
- 持久化任务状态。

这个原型故意很小，因为它的作用不是模拟完整 Claude Code，而是帮助理解 Harness 组件如何拼起来。

## 6. 继续扩展

掌握最小 Harness Agent 后，可以继续扩展三个方向：

- **DeepResearch**：把搜索、反思和报告输出加入 Agent Loop。
- **Memory System**：引入向量库、图数据库或 Mem0 风格的记忆生命周期。
- **Claw 类产品**：在 Code Agent 外层加 IM 通信、定时任务、容器调度和多会话管理。
