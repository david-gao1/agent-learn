# 什么是 Code Agent：它为什么用代码作为行动方式

## 一、没有 Code Agent 时，问题是什么

传统工具调用 Agent 通常把每个外部能力包装成一个工具。

比如查询天气有天气工具，计算数据有计算工具，搜索网页有搜索工具，查询数据库有数据库工具。

这种方式清晰，但也会遇到一个问题：工具越多，模型越难选择；任务越灵活，工具越难提前定义。

更麻烦的是，有些任务用单次工具调用非常低效。

书里举了一个简单例子：计算 1 加到 100。

如果 Agent 只有一个 `calc()` 加法工具，ReAct Agent 可能需要一轮轮调用工具，把每次计算结果放回上下文。这样既慢，又容易因为长程步骤出错。

人类程序员不会这么做。我们会写一个循环，一次性完成计算。

Code Agent 的思路，就是让模型也用代码表达行动。

## 二、Code Agent 的定义

Code Agent 是一种以生成并执行代码为核心行动方式的 Agent 范式。

它不是指 Cursor、Claude Code 这类辅助编程产品本身，而是指一种通用 Agent 设计方式：模型不直接调用大量细碎工具，而是生成 Python、Shell 或其他代码，让执行器运行代码并返回结果。

传统 ReAct Agent 更像是在选择工具。

Code Agent 更像是在编写一段小程序，让这段程序完成一组动作。

它把“工具调用”提升成了“代码执行”。

## 三、Code Agent 的核心机制

Code Agent 的核心机制，是把任务逻辑放进代码里执行。

以 1 加到 100 为例，模型可以生成一段循环：

```python
result = 0
for i in range(1, 101):
    result = calc(result, i)
print(result)
```

执行器运行这段代码，一次性得到结果。

这和 ReAct 一轮轮调用工具不同。

ReAct 的循环发生在模型对话层。每一步都要经过模型判断、工具调用、观察返回。

Code Agent 的循环可以发生在代码运行层。只要逻辑明确，代码执行器可以在一次行动中完成多步计算、文件处理或系统诊断。

所以，Code Agent 的优势不只是“会写代码”，而是把可程序化的步骤从模型上下文里卸载到执行环境里。

## 四、Code Agent 的关键组成

Code Agent 至少需要四个组成部分。

第一是代码生成能力。模型必须能根据任务生成可运行、可理解的代码。

第二是执行环境。系统需要一个 Python、Shell、容器或沙箱来运行代码。

第三是安全边界。代码执行比普通工具调用更危险，必须限制文件权限、网络权限、命令范围和运行时间。

第四是结果回传机制。执行结果不能只是原始输出，还要被整理成模型下一步能使用的观察。

如果缺少安全边界，Code Agent 很容易变成风险源。

如果缺少结果整理，执行输出又会变成新的上下文噪声。

## 五、Code Agent 解决了什么，没解决什么

Code Agent 解决的核心问题，是让 Agent 能高效处理可程序化任务。

它适合数据处理、运维诊断、文件批量操作、图表生成、代码分析等场景。

它能减少多轮工具调用，把重复步骤交给代码执行器完成。

它也能提升灵活性。开发者不必为每个小动作都提前封装一个工具，模型可以临时组合逻辑。

但 Code Agent 也不是万能的。

它依赖模型写代码的能力。如果代码错了，执行结果也会错。

它需要更强的安全治理。Shell 命令、文件写入、网络请求都可能造成真实影响。

它也不适合所有任务。开放式研究、价值判断、业务决策，仍然需要规划、反思和人工审核。

Code Agent 解决的是行动表达方式，不自动解决上下文管理、任务规划和风险控制。

## 六、一句话收束

Code Agent 的本质，是把适合程序化执行的步骤从模型对话中移到代码执行环境里，用代码作为更高效、更灵活的行动方式。
