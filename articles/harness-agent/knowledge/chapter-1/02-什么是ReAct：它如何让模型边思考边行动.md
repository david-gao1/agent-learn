# 什么是 ReAct：它如何让模型边思考边行动

## 一、没有 ReAct 时，问题是什么

大模型本身擅长生成文本，但它并不天然知道实时世界发生了什么。

如果用户问“今天北京天气怎么样”，模型仅靠训练知识无法保证答案准确。

Function Calling 让模型可以调用工具，解决了一部分问题。模型可以选择 `get_weather` 这样的函数，构造参数，拿到工具返回结果，再生成回答。

但早期 Function Calling 也有局限。

一方面，并不是所有模型都原生支持工具调用。另一方面，模型为什么选择某个工具、下一步如何推理，对开发者来说仍然比较黑盒。

当任务变复杂时，模型还需要多轮观察和调整，而不是一次性调用一个函数就结束。

ReAct 就是在这个背景下流行起来的。

## 二、ReAct 的定义

ReAct 是 Reasoning and Acting 的缩写，意思是让模型在推理和行动之间交替进行。

它不是一个具体产品，而是一种 Agent 构建范式。

在 ReAct 中，模型会先推理当前问题，然后决定是否调用工具，接收工具观察结果，再基于新的观察继续推理。

它的基本循环是：

```text
Reason -> Act -> Observe -> Reason
```

Reason 是模型的推理过程。

Act 是模型选择工具并发起行动。

Observe 是系统把工具结果返回给模型。

下一轮 Reason 会基于新的上下文继续判断。

## 三、ReAct 的核心机制

ReAct 的关键，是把一次性回答变成多轮上下文演化。

模型不再只基于用户原始问题回答，而是可以边做边看。

以代码仓库分析为例。用户让 Agent 分析项目架构，模型一开始不知道文件结构。

第一轮 Reason 会判断：我需要先看目录。

Act 调用文件列表工具。

Observe 返回目录结构。

第二轮 Reason 会基于目录判断：README 和配置文件可能最重要。

Act 读取这些文件。

Observe 返回内容。

后续模型再继续追踪入口文件、测试命令、核心模块。

在这个过程中，每次工具观察都会改变下一轮上下文。

所以，ReAct 不只是工具调用格式，而是一种让模型通过行动不断更新工作记忆的机制。

## 四、ReAct 的关键组成

ReAct 至少包含五个组成部分。

第一是任务目标。模型必须知道当前要完成什么，否则推理会漂移。

第二是工具列表。模型需要知道有哪些工具可用，每个工具能做什么。

第三是推理格式。系统通常会通过提示词约定 Reason、Action、Observation 的交互格式。

第四是工具执行器。模型本身不真正执行工具，外部系统负责接收工具调用并返回结果。

第五是上下文更新。工具返回结果必须进入下一轮模型输入，否则行动不会影响推理。

这些组成部分共同构成 ReAct 的循环。

## 五、ReAct 解决了什么，没解决什么

ReAct 解决了一个重要问题：模型可以通过外部行动获得新信息，并把新信息纳入后续推理。

它降低了构建 Agent 的门槛。即使模型没有原生 Function Calling，只要能遵循提示词格式，也能实现类似工具调用的行为。

它也提高了可解释性。开发者能看到模型在每一步为什么行动、拿到了什么观察、下一步如何继续。

但 ReAct 也有明显局限。

它依赖模型的指令遵循能力。如果模型不按格式输出，工具调用就会失败。

它容易变慢。复杂任务需要多轮模型调用，每一轮都有成本和延迟。

它缺乏全局规划。模型可能被局部观察牵着走，陷入低效路径。

它也容易造成上下文膨胀。每轮推理、工具调用和观察结果都会占用上下文窗口。

因此，后续才会出现计划模式、反思模式、CodeAct、SubAgent、上下文压缩等更复杂的 Harness 设计。

## 六、一句话收束

ReAct 的本质，是让模型通过“推理、行动、观察”的循环，把外部世界的反馈不断写入自己的工作记忆。
