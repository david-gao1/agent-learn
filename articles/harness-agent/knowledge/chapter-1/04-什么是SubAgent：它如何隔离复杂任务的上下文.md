# 什么是 SubAgent：它如何隔离复杂任务的上下文

## 一、没有 SubAgent 时，问题是什么

单个 Agent 可以完成很多任务，但复杂任务会让它的上下文迅速变脏。

一个长任务里会出现用户意图、计划、工具说明、命令输出、错误日志、搜索结果、临时假设、历史对话等大量信息。

这些信息并不是都没用。

问题在于，它们的作用范围不同。

有些信息只对某个局部子任务有用，比如测试失败日志。

有些信息只在探索阶段有用，比如临时猜测。

有些信息才应该进入主线，比如最终结论、证据位置和风险判断。

如果所有过程信息都留在一个 Agent 的上下文里，模型会越来越难抓住主线，上下文窗口也会被快速消耗。

SubAgent 就是在这种情况下出现的。

## 二、SubAgent 的定义

SubAgent，也叫子 Agent 或专家 Agent，是在一个更大的 Agent 系统中，负责某个局部任务的独立 Agent。

典型结构是：一个主 Agent 负责拆解任务、分配工作、整合结果；多个 SubAgent 负责各自的局部探索或专项处理。

从分工角度看，SubAgent 是专项执行者。

从上下文工程角度看，SubAgent 更重要的作用是隔离上下文。

它把局部任务的试错过程、工具输出和中间判断留在自己的上下文里，只把摘要、证据和结论交还给主 Agent。

## 三、SubAgent 的核心机制

SubAgent 的机制可以理解成一次受控委托。

主 Agent 首先识别出一个可以独立处理的子任务。

然后，它把目标、边界和必要背景交给 SubAgent。

SubAgent 在自己的上下文窗口里完成探索：读文件、查日志、调用工具、验证假设、整理发现。

最后，SubAgent 返回压缩后的结果，而不是完整过程。

以代码仓库分析为例，主 Agent 的目标是写架构说明。

它可以把“分析测试体系”“寻找 API 入口”“梳理数据模型”拆给不同 SubAgent。

每个 SubAgent 只返回关键发现、证据文件、风险判断。

主 Agent 不需要背负所有测试日志、目录扫描和临时猜测，只需要整合结果。

这就是上下文隔离。

## 四、SubAgent 的关键组成

SubAgent 系统至少包含四个组成部分。

第一是主 Agent。它负责理解总目标、拆解任务、分派子任务、整合结果。

第二是子任务边界。每个 SubAgent 必须有清楚目标，知道自己要交付什么。

第三是独立上下文窗口。SubAgent 的中间过程不直接污染主 Agent。

第四是结果协议。SubAgent 返回的内容应该是摘要、证据、风险、建议，而不是完整聊天记录。

如果子任务边界不清楚，SubAgent 会和主 Agent 互相干扰。

如果结果协议不清楚，主 Agent 仍然无法可靠整合。

## 五、SubAgent 解决了什么，没解决什么

SubAgent 解决的核心问题，是复杂任务中的上下文污染。

它能让局部探索在局部完成，让主 Agent 保持干净的上下文。

它也能带来分工收益。不同 SubAgent 可以关注不同领域、工具或任务阶段。

但 SubAgent 不等于越多越好。

拆得太细，会增加调度成本。主 Agent 要等待结果、处理冲突、判断哪个结论可信。

SubAgent 也可能丢信息。它把过程压缩成摘要时，可能漏掉主 Agent 后续需要的细节。

它还可能制造不一致。不同 SubAgent 可能对同一份代码或同一个风险给出不同判断。

所以，SubAgent 适合局部目标清楚、中间噪声多、结果可以摘要的任务。

如果主 Agent 的上下文还很干净，任务也不长，强行拆 SubAgent 只是制造复杂度。

## 六、一句话收束

SubAgent 的本质，是把复杂任务中的局部探索放进独立上下文，只把可整合的结果交回主线。
