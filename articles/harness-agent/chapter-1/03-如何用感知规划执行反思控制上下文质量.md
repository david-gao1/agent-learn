# 如何用感知、规划、执行、反思控制上下文质量

很多人会把上下文理解成“给模型的资料”。

资料越多，模型越聪明；背景越全，回答越准。这个判断在简单任务里有时成立，但放到复杂任务里会出问题。

**上下文不是资料包，而是一份需要持续治理的工作记忆。**

真正困难的地方，不是把更多内容放进窗口，而是让进入窗口的信息始终贴近目标、顺序合理、噪声可控、可以被修正。

这就是感知、规划、执行、反思四个模块的价值。

## 一、表面问题是资料不够，真正问题是上下文质量不可控

假设任务是：让 Agent 分析一个代码仓库，并写出架构说明。

最粗暴的做法是把所有文件都交给模型。这样看起来信息最全，但实际很容易失败。

模型会被细节淹没：缓存文件、构建产物、边缘模块、历史脚本、测试样例，都可能和核心架构混在一起。

上下文一旦没有结构，模型就不知道该先理解目标，还是先读文件；该相信 README，还是相信启动脚本；该关注入口，还是关注工具函数。

所以问题不是“资料还不够多”，而是“上下文质量没有被控制”。

感知、规划、执行、反思这四个模块，本质上就是控制上下文质量的四道工序。

## 二、四个模块不是功能列表，而是一条治理链

感知解决入口问题。

用户说“帮我看看这个项目”，这句话并不精确。他可能想知道怎么启动，也可能想找架构风险，也可能想写 README，也可能想判断这个项目是否值得学习。

感知的任务，是把自然语言请求转成结构化意图。没有这一步，后面的工具调用很容易跑偏。

规划解决顺序问题。

对代码仓库分析来说，合理顺序不是随便读文件，而是先列目录，再读 README，再看配置，再找入口，再看测试，最后抽核心模块。

规划的作用不是显得严谨，而是控制上下文生成路径，避免模型被局部信息牵着走。

执行解决事实来源问题。

它让 Agent 接触外部环境：读文件、跑命令、查资料、调用 API。每次执行都会产生观察结果。

但执行结果只是原料，不等于上下文。系统还要决定哪些观察进入当前窗口，哪些只保留引用，哪些应该丢弃。

反思解决质量校验问题。

现在的信息够吗？有没有关键文件没读？命令失败是否影响结论？是否只看到了表层目录，没有理解调用链？这些问题都属于反思。

反思不是简单挑错，而是检查上下文是否足以支撑下一步判断。

## 三、判断四个模块有没有真正发挥作用

可以用四个问题检查。

第一，感知有没有把模糊请求变成明确任务？

如果用户说“看看项目”，系统至少要区分：是启动说明、架构理解、风险评估，还是学习路线。否则后面的上下文都会没有方向。

第二，规划有没有决定信息进入上下文的顺序？

好的规划会减少无效阅读。它不是列一堆步骤，而是解释为什么先读这个、后读那个。

第三，执行有没有产生可用观察？

执行不是为了多跑命令，而是为了拿到能改变判断的事实。无关输出、重复日志、失败命令，都需要被处理，而不是原样堆进上下文。

第四，反思有没有修正信息缺口？

如果 Agent 写报告前没有问“证据够不够、结论是否过早、还有没有未验证假设”，它就容易把半成品上下文当成事实。

这四个问题连起来，就是上下文质量控制链：目标是否清楚，路径是否合理，事实是否可靠，结论是否被校验。

## 四、四模块也不是越完整越好

不是每个任务都需要完整的感知、规划、执行、反思链。

如果任务很小，比如总结一段已经给出的文本，感知和规划可以很轻。强行套完整流程，只会增加成本。

如果任务风险低，反思也不必复杂。一次简单格式转换，不需要专门设计评审 Agent。

四模块真正适合的是长链条、开放式、信息不完整、结果需要可信的任务。

比如代码仓库分析、DeepResearch、运维诊断、ChatBI 查询。这些任务里，错误的入口理解会导致全程跑偏，错误的阅读顺序会制造噪声，错误的工具观察会传播到结论，缺少反思会让半成品被包装成答案。

所以，四模块不是为了显得架构完整，而是为了在复杂任务里控制上下文质量。

一句话收束：**感知、规划、执行、反思不是功能清单，而是让上下文从模糊请求生长为可信工作记忆的治理链。**
