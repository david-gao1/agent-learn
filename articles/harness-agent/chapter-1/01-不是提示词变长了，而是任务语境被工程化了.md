# 不是提示词变长了，而是任务语境被工程化了

中心问题：上下文工程到底比提示词工程多了什么？

## 1. 一个常见误解

很多人第一次听到上下文工程，会把它理解成“写更长的提示词”。这很自然，因为从表面看，提示词确实变长了。

但长度不是关键。真正的变化是：输入模型的信息不再只是人的一句指令，而是一组经过采集、筛选、组织和更新的任务语境。

提示词工程关注“怎么说”。上下文工程关注“模型此刻应该知道什么”。

## 2. 购车问题为什么是好例子

书中用“理想 L6 怎么样”作为例子。这个问题本身太短，也太空。模型可以回答很多参数，但很难判断哪些参数对提问者真正重要。

一旦补充通勤距离、家庭成员、预算、持有周期、试驾反馈、电池供应商等信息，问题就变了。它不再是问一辆车“好不好”，而是在问这辆车是否适合一个具体人的具体约束。

这里的关键不是把问题写成一篇小作文，而是把决策所需的信息结构化了。

## 3. Prompt 仍然是静态输入

提示词工程可以把一句话改得更清楚、更有角色感、更有格式约束。它能提升单次回答质量。

但它通常仍然假设：人已经知道要放什么信息，并且一次性把信息放进去。

这在简单任务里够用。比如让模型改一段文案、总结一篇文章、生成一个表格。任务边界清楚，外部信息少，过程不需要变化。

复杂任务不是这样。复杂任务会暴露新的问题：信息不全、目标变化、工具返回新结果、早期判断需要修正、上下文窗口会被噪声填满。

## 4. Context 是动态工作记忆

上下文工程把模型输入看成“工作记忆”。这份工作记忆不是越多越好，而是要在每一步都保持相关、可靠、结构清楚。

它至少包含五类信息：用户意图、任务边界、外部知识、工具观察、历史记忆。

这五类信息的来源不同，时效不同，可信度不同。把它们简单拼在一起，会制造噪声。上下文工程要做的是选择、排序、压缩和更新。

所以，上下文工程不是“多给模型一些背景”。它是在设计模型的注意力环境。

## 5. 分界线在哪里

一个粗略判断是：如果信息仍然由人一次性整理好交给模型，这更接近提示词工程。

如果系统能够持续识别意图、调用工具、获取观察、压缩历史、加载相关知识，并把这些内容组织成下一轮输入，这就进入了上下文工程。

前者是一次表达。后者是一个过程。

## 6. 为什么这会通向 Harness

一旦把上下文看成动态工作记忆，就会自然需要一套外部系统来维护它。

这个系统需要工具来获取信息，需要记忆来保存信息，需要 Skills 来按需加载专家经验，需要 Agent Loop 来驱动下一步行动。

这些“模型之外”的工程能力，就是 Harness 思路的入口。

## 7. 这一篇的结论

上下文工程不是提示词工程的加强版，而是任务语境的工程化。

它关心的不是一句话怎么写得更聪明，而是模型在每一步工作时，是否拥有刚好足够、结构清楚、噪声可控的工作记忆。
