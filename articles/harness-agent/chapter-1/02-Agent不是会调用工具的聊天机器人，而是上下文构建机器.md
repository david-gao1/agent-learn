# Agent 不是会调用工具的聊天机器人，而是上下文构建机器

中心问题：为什么上下文工程天然需要 Agent？

## 1. 工具调用只是表象

很多人理解 Agent，会先想到工具调用。模型能查天气、搜网页、读文件、执行命令，所以它变成了 Agent。

这当然没错，但还不够深。工具调用只是外在动作。更重要的是，每一次动作都会改变模型下一步能看到的上下文。

从这个角度看，Agent 的核心不是“会用工具”，而是“能通过行动持续构建上下文”。

## 2. ReAct 的真正启发

ReAct 的循环可以简化成四步：推理、行动、观察、再推理。

Reason 决定下一步要知道什么。Act 调用工具去获得信息。Observe 把工具结果带回系统。下一轮 Reason 基于新的上下文继续判断。

这不是普通聊天。普通聊天主要在已有文本里生成回复。ReAct 则让模型用外部世界刷新自己的工作记忆。

## 3. 静态上下文为什么不够

复杂任务常常无法在开始时就知道全部信息。

比如让 Agent 分析一个代码仓库。它一开始不知道项目结构，不知道入口文件，不知道测试命令，也不知道哪些模块最关键。

如果让人一次性把所有文件塞进上下文，成本高，噪声大，还容易超过窗口。更好的方式是让 Agent 先列文件，再读 README，再读入口，再根据发现继续探索。

这里，Agent 每一步都不是为了“完成一个动作”而动作，而是在为下一步推理准备更好的上下文。

## 4. Agent 是上下文演化的执行者

上下文工程需要几个能力：识别缺什么信息，决定去哪拿，调用工具获取，把结果整理成可用上下文，再判断是否继续。

这些动作无法靠一条提示词完成。它们需要循环，需要状态，需要工具，需要终止条件。

这就是 Agent 出场的原因。Agent 是上下文工程的执行载体。

## 5. 工具结果不能原样堆进去

Agent 调工具之后，还会遇到一个新问题：工具结果经常很长、很脏、很局部。

搜索结果可能重复。命令输出可能包含无关日志。代码阅读可能产生大量细节。把这些内容全部塞回模型，会让上下文变胖，也会让模型分心。

所以 Agent 不只是收集信息，还必须筛选信息。好的 Agent Loop 应该把观察转化成下一步真正需要的上下文片段。

## 6. 这一视角改变了设计重点

如果把 Agent 看成工具调用器，设计重点会落在“工具多不多、参数准不准”。

如果把 Agent 看成上下文构建机器，设计重点会变成“每一步是否让下一步更清楚、更接近目标、更少噪声”。

这会影响很多工程选择：工具输出要截断，历史要压缩，任务状态要持久化，复杂子任务要交给 SubAgent 隔离。

## 7. 这一篇的结论

Agent 的价值不只是能行动，而是能通过行动让上下文不断变好。

上下文工程需要 Agent，因为复杂任务的有效上下文不是一次写出来的，而是在推理、行动、观察、更新的循环中长出来的。
