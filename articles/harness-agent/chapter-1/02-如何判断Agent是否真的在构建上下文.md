# 如何判断 Agent 是否真的在构建上下文

很多人理解 Agent，会先想到工具调用。

模型能查天气、搜网页、读文件、执行命令，于是它就从聊天机器人变成了 Agent。这个说法没错，但只讲到表层。

**Agent 的关键不只是能调用工具，而是能通过行动降低下一步判断的不确定性。**

如果工具调用没有改变下一步判断，它只是把聊天窗口接上了几个按钮。真正有价值的 Agent，会让上下文随着任务推进变得更准确、更聚焦、更可行动。

## 一、问题不在有没有工具，而在工具是否改变判断

普通聊天主要在已有文本里生成回复。用户给什么，模型就基于这些内容回答。

Agent 不一样。它可以先判断自己缺什么信息，再调用工具去获得信息，然后把观察结果带回下一轮推理。

ReAct 的启发就在这里：Reason 决定下一步要知道什么，Act 去外部世界拿信息，Observe 把结果带回来，下一轮 Reason 基于新上下文继续判断。

所以，工具调用只是外在动作。更重要的是，工具结果是否让模型下一步更清楚。

比如让 Agent 分析一个陌生代码仓库。它一开始并不知道项目结构、入口文件、测试命令、核心模块在哪里。

如果它只是执行 `ls`，然后把所有结果原样塞给模型，这还很浅。

如果它能根据 `ls` 的结果判断先读 README，再看配置，再找入口文件，再根据入口追踪核心模块，这时工具调用才真正变成了上下文构建。

## 二、真正的问题是上下文能不能持续演化

复杂任务的有效上下文，通常不是一开始就能准备好的。

代码仓库分析如此，线上问题排查也是如此。你一开始只知道“系统慢”，但不知道慢在网关、服务、数据库、缓存，还是第三方接口。

这时，如果把 Agent 当成工具调用器，设计重点会落在“给它多少工具”。但如果把 Agent 当成上下文构建器，设计重点会变成“它能不能逐步缩小不确定性”。

一次查询日志，应该帮助它判断下一步查哪个服务；一次读文件，应该帮助它判断哪个模块是入口；一次失败命令，应该帮助它修正假设，而不是机械重试。

Agent 的每一步行动，都应该让上下文更接近任务真相。

这也是它和长提示词的分界。

长提示词把已有信息放得更完整。Agent 则在任务过程中不断发现新信息，并调整后续路径。

## 三、判断一个 Agent 是否真的在构建上下文

可以用三个问题判断。

第一，它是否带着信息缺口行动？

好的 Agent 调工具之前，应该隐含一个问题：我现在缺什么信息？是缺入口文件、缺错误日志、缺指标定义，还是缺用户意图？

如果没有信息缺口，只是为了“展示能调用工具”而调用工具，那不是上下文构建。

第二，它是否把观察转成结构化判断？

工具结果通常很脏。搜索结果会重复，命令输出有无关日志，代码文件有大量细节。把它们全部塞回模型，只会让上下文变胖。

好的 Agent 会把观察转成下一步需要的判断：这个文件是入口；这个错误来自权限；这个模块只是适配层；这个结果需要继续验证。

第三，它是否会因为新观察改变路径？

如果 Agent 的计划完全不受观察影响，它只是在执行固定脚本。

真正的 Agent 会根据观察调整路径。发现 README 过期，就转去看启动脚本；发现测试命令失败，就读配置；发现代码里没有数据库访问，就重新判断架构形态。

能修正路径，才说明上下文在演化。

## 四、Agent 也会制造新的上下文问题

不是所有任务都需要 Agent。

如果任务是单轮、信息完整、低风险的，直接提示词就够了。让模型改写一段文字、总结一段材料、生成固定格式内容，不需要工具循环。

引入 Agent 后，系统会变复杂。你要处理工具失败、输出截断、上下文压缩、循环终止、安全边界、错误传播。

Agent 还容易制造噪声。它越能行动，越可能收集太多信息。工具越多，错误路径也越多。

所以，Agent 的价值不在于“行动更多”，而在于“每次行动都让下一步判断更好”。

如果一个 Agent 只是不断调用工具，却没有让上下文更清楚，它不是更智能，只是更忙。

真正值得构建的 Agent，是能把行动转化为更好的工作记忆。

一句话收束：**Agent 是否有价值，不看它调用了多少工具，而看它是否通过工具观察持续改善上下文。**

## 这篇文章还需要怎么优化

这篇文章的主线是“行动是否降低不确定性”。后续修改不要扩展 Agent 分类，也不要提前讲完整 Harness。

如果要加深，只补一个最小循环图：`信息缺口 -> 工具行动 -> 观察 -> 结构化判断 -> 路径修正`。
