# 如何判断 Agent 是否真的在构建上下文

很多人判断一个系统是不是 Agent，第一反应是看它会不会调用工具。

能查天气、能搜网页、能读文件、能执行命令，好像就已经从聊天机器人变成了 Agent。

这个判断不算错，但太浅。

**Agent 的关键，不是它能不能行动，而是它的行动有没有让下一步判断更清楚。**

如果工具调用没有改变后续判断，它只是把聊天窗口接上了几个按钮。真正有价值的 Agent，会通过行动不断降低不确定性，让上下文越来越接近任务真相。

这才是 Agent 和普通工具调用之间的分界。

## 一、问题不在有没有工具，而在工具是否改变判断

普通聊天主要在已有文本里生成回复。用户给什么，模型就基于这些内容回答。

Agent 不一样。它可以先判断自己缺什么信息，再调用工具去获得信息，然后把观察结果带回下一轮推理。

这也是 ReAct 真正有价值的地方。

Reason 不是单纯“思考一下”，而是在判断下一步缺什么。Act 不是随便调用工具，而是去外部世界补信息。Observe 不是把结果贴回来就结束，而是把观察变成下一轮判断的依据。

如果这条链断了，工具调用就只是动作表演。

比如让 Agent 分析一个陌生代码仓库。它一开始并不知道项目结构、入口文件、测试命令、核心模块在哪里。

如果它只是执行 `ls`，然后把所有输出原样塞进上下文，这还只是浅层工具调用。

如果它能根据 `ls` 的结果判断先读 README，再看配置文件，再找入口文件，再根据入口追踪核心模块，这时工具调用才开始变成上下文构建。

因为每一次观察，都在改变下一步该看什么。

## 二、真正的问题是 Agent 能不能缩小不确定性

复杂任务一开始通常不是缺答案，而是缺判断条件。

代码仓库分析如此，线上故障排查也是如此。你一开始只知道“系统慢”，但不知道慢在网关、业务服务、数据库、缓存，还是第三方接口。

这时，如果把 Agent 当成工具调用器，设计重点会落在“给它多少工具”。

但如果把 Agent 当成上下文构建器，设计重点会变成另一个问题：**它能不能通过每一步行动缩小不确定性。**

一次查询日志，应该帮助它判断下一步查哪个服务。

一次读配置，应该帮助它判断系统如何启动。

一次失败命令，应该帮助它修正假设，而不是机械重试。

一次搜索结果，应该帮助它形成新的验证方向，而不是把网页摘要全部堆进上下文。

所以，Agent 的行动不应该只是“做了什么”，还要回答“这个动作让后续判断发生了什么变化”。

如果行动之后，下一步判断没有更清楚，那这个行动对上下文构建没有贡献。

## 三、判断一个 Agent 是否真的在构建上下文

可以用三个问题判断。

第一，它是不是带着信息缺口行动？

好的 Agent 调工具之前，应该隐含一个问题：我现在缺什么信息？

分析代码仓库时，可能缺入口文件；排查故障时，可能缺错误日志；做 ChatBI 时，可能缺指标口径；做研究任务时，可能缺可信来源。

如果没有信息缺口，只是为了展示“我能调用工具”，那不是上下文构建。

第二，它是不是把观察转成结构化判断？

工具结果通常很脏。命令输出有无关日志，搜索结果会重复，代码文件有大量细节，数据库查询可能只反映局部状态。

好的 Agent 不会把这些结果原样堆回上下文，而会把观察转成判断：这个文件是入口；这个错误来自权限；这个模块只是适配层；这个结论还缺证据；这个方向应该放弃。

上下文不是工具输出的集合，而是被整理过的判断材料。

第三，它是不是会因为新观察改变路径？

如果 Agent 的计划完全不受观察影响，它只是在执行固定脚本。

真正的 Agent 会根据观察调整路径。发现 README 过期，就转去看启动脚本；发现测试命令失败，就读配置；发现代码里没有数据库访问，就重新判断系统形态。

能改变路径，才说明上下文在演化。

如果一个 Agent 从头到尾只是按预设步骤跑完，即使它调用了很多工具，也未必真的在构建上下文。

## 四、Agent 也会制造新的上下文问题

不是所有任务都需要 Agent。

如果任务是单轮、信息完整、低风险的，直接提示词就够了。比如改写一段文字、总结一段材料、生成固定格式内容，不需要工具循环。

引入 Agent 后，系统会变复杂。你要处理工具失败、输出截断、上下文压缩、循环终止、安全边界、错误传播。

Agent 还容易制造噪声。它越能行动，越可能收集太多信息。工具越多，错误路径也越多。

所以，Agent 的价值不在于“行动更多”，而在于“每次行动都让下一步判断更好”。

如果一个 Agent 只是不断调用工具，却没有让上下文更清楚，它不是更智能，只是更忙。

真正值得构建的 Agent，是能把行动转化为更好的工作记忆。

判断一个 Agent 是否真的在构建上下文，不看它调用了多少工具，而看三件事：它是否带着信息缺口行动，是否把观察整理成判断，是否会因为新观察修正路径。

如果答案是否定的，它只是一个会动的聊天机器人。

如果答案是肯定的，它才开始接近真正的上下文构建系统。
