# 上下文不是堆出来的：感知、规划、执行、反思的生命周期

中心问题：上下文构建为什么需要感知、规划、执行、反思四个模块？

## 1. 上下文不是资料包

很多人会把上下文理解成“给模型的资料”。资料越多，模型越聪明。

这其实很危险。资料太多会制造噪声，资料顺序不对会打断推理，资料缺少结构会让模型不知道该信什么。

上下文更像一个生命周期。它要被识别、组织、验证和修正。

## 2. 用一个任务贯穿

假设任务是：让 Agent 分析一个代码仓库，并写出结构说明。

如果直接把全部文件交给模型，模型可能被细节淹没。更合理的方式，是让系统按阶段构建上下文。

这正好对应第 1 章提到的四个模块：感知、规划、执行、反思。

## 3. 感知：先弄清用户到底要什么

感知模块解决入口问题。用户说“帮我看看这个项目”，这句话并不精确。

他可能想知道项目怎么启动，也可能想找架构问题，也可能想写 README，也可能想判断是否值得继续学习。

感知的目标，是把自然语言请求转成结构化任务意图。没有这一步，后面的工具调用很容易跑偏。

## 4. 规划：决定上下文生成顺序

规划模块决定先看什么、后看什么。

对代码仓库分析来说，合理顺序可能是：先列文件，再读 README，再看配置，再找入口，再看测试，最后抽核心模块。

规划不是为了仪式感。它的作用是控制上下文生成路径，避免模型被局部信息牵着走。

## 5. 执行：用工具获得真实观察

执行模块负责让 Agent 接触外部环境。

它可以读文件、跑命令、查搜索、调用 API。每次执行都会返回观察结果，这些结果构成新的上下文材料。

但执行的结果还不是上下文本身。它只是原料。系统还需要筛选哪些结果进入下一轮推理。

## 6. 反思：检查上下文有没有错漏

反思模块负责问几个问题：现在的信息够吗？有没有关键文件没读？有没有命令失败？有没有只看到了表层结构，没有理解核心流程？

反思不是简单挑错，而是检查上下文质量。

没有反思，Agent 很容易在一个看似合理但其实不完整的路径上继续前进。

## 7. 四个模块其实是一条链

感知决定目标，规划决定路径，执行产生观察，反思修正方向。

这四个模块不是并列功能，而是一条上下文生命周期链。

每轮循环之后，系统都会拥有比上一轮更贴近任务目标的上下文。

## 8. 这一篇的结论

上下文不是堆出来的，而是沿着任务生命周期生长出来的。

感知、规划、执行、反思四个模块的价值，是让上下文从模糊请求逐步变成可推理、可行动、可验证的工作记忆。
