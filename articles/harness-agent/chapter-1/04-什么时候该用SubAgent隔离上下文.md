# 什么时候该用 SubAgent 隔离上下文

很多人解释 SubAgent，会说它是为了分工。

主 Agent 负责调度，子 Agent 负责专项任务。听起来像一个团队，有人负责测试，有人负责 API，有人负责数据模型。

这个说法没错，但还不够深。

**SubAgent 的关键，不是多几个执行者，而是把局部探索的噪声锁在局部。**

从上下文工程看，SubAgent 首先是一种上下文隔离手段，其次才是分工手段。

如果没有上下文污染问题，SubAgent 很可能只是把系统拆复杂了。

## 一、问题不在任务太多，而在上下文太脏

一个复杂任务会产生大量中间内容：用户原始意图、计划、工具说明、命令输出、错误日志、搜索结果、临时假设、历史对话。

这些内容不是都没用。问题在于，它们的作用范围不同。

有些信息只对某个子任务有用，比如测试失败日志；有些信息只在探索阶段有用，比如临时猜测；有些信息对主任务很重要，比如最终风险判断。

如果所有内容都堆在主 Agent 的上下文里，主线会被过程噪声淹没。

这时问题不是模型不够聪明，而是它工作的上下文环境坏了。

SubAgent 要解决的，不是“主 Agent 太忙”，而是“主 Agent 不该背负所有局部过程”。

## 二、真正的问题是局部探索该不该进入主上下文

让 Agent 分析一个大型代码仓库时，主 Agent 的目标可能是写出架构说明。

但这个目标下面有很多局部探索：测试体系怎么跑，API 入口在哪里，数据模型怎么设计，权限逻辑在哪里，部署方式是什么。

如果主 Agent 亲自完成所有探索，它的上下文会塞满中间过程。

读测试产生的失败日志、找入口产生的目录扫描、看数据模型产生的字段细节，都会留在主上下文里。

这些信息对局部任务有用，但对主线整合未必有用。

SubAgent 的做法，是把一段局部探索放进独立窗口。

主 Agent 只下发清楚的子任务。SubAgent 自己读文件、试命令、整理证据、排除噪声。完成后，它只把关键发现、证据位置和风险判断返回。

这样，主 Agent 保留的是目标、任务分配和结果整合，而不是所有试错过程。

SubAgent 的价值，就在于让局部复杂性不要污染全局判断。

## 三、判断什么时候该拆 SubAgent

可以用四个问题判断。

第一，这个子任务有没有独立目标？

如果只是主任务里的一个小步骤，比如读一个文件，就不需要 SubAgent。如果它能形成独立结论，比如“测试体系是否可靠”，才值得隔离。

第二，它是否会产生大量中间噪声？

如果子任务需要多轮搜索、读很多文件、尝试命令、处理错误日志，它很适合交给 SubAgent。因为这些过程信息不应该污染主上下文。

第三，它的结果能否被压缩成清晰交付物？

好的 SubAgent 输出应该是摘要、证据、风险、建议，而不是把自己的完整聊天记录搬回来。

如果子任务无法形成清楚交付物，拆出去后主 Agent 仍然无法整合。

第四，它是否有独立工具或知识边界？

测试分析、API 入口分析、数据模型分析，通常需要不同关注点。它们适合分开。

反过来，如果几个步骤高度耦合，强拆只会增加协调成本。

真正适合 SubAgent 的任务，通常同时满足三个条件：局部目标清楚，中间噪声多，最终结果可以被摘要。

## 四、SubAgent 也会制造复杂度

SubAgent 不是越多越好。

拆得太细，会引入新的协调成本。主 Agent 要分派任务、等待结果、处理冲突、判断哪个子 Agent 的结论可信。

SubAgent 还可能丢信息。它把过程压缩成摘要时，可能漏掉主 Agent 后面需要的细节。

它也可能制造不一致。不同 SubAgent 对同一份代码、同一个概念、同一个风险的判断可能不一致，主 Agent 必须有整合和仲裁能力。

所以，SubAgent 不是为了让系统看起来更像团队，而是为了在上下文污染已经影响主线判断时，主动隔离局部探索。

如果主 Agent 的上下文还很干净，任务也不长，强行拆 SubAgent 只是把简单问题变成调度问题。

判断什么时候该用 SubAgent，不看任务能不能被分工，而看局部探索的噪声是否已经不该进入主上下文。

如果没有上下文污染，先不要拆。

如果局部探索会产生大量过程信息，而主 Agent 只需要结论、证据和风险，那才值得拆。
