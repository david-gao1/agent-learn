# SubAgent 的真正价值：不是更多人手，而是更干净的上下文

中心问题：多 Agent 的本质是分工，还是上下文隔离？

## 1. 常见说法只讲了一半

很多资料解释多 Agent，会说因为工具太多，所以要分专家 Agent。

这个说法有道理。一个 Agent 面对太多工具，确实容易选错，也容易被复杂说明干扰。

但从上下文工程看，更深的问题不是“人手不够”，而是“上下文太脏”。

## 2. 单 Agent 为什么会被污染

一个复杂任务会产生很多中间内容：用户原始意图、计划、工具说明、命令输出、错误日志、搜索结果、临时假设、历史对话。

这些内容有些重要，有些只是过程噪声。有些只对子任务有用，对主任务反而是干扰。

如果全部留在一个 Agent 的上下文里，模型会越来越难抓住主线。

## 3. 上下文污染的两个后果

第一个后果是注意力分散。模型看到太多不相关细节，会把局部信息误当成关键线索。

第二个后果是窗口耗尽。任务还没完成，上下文已经塞满，中间观察和历史轨迹开始挤掉真正重要的信息。

这时问题不是模型不够聪明，而是它工作的上下文环境坏了。

## 4. SubAgent 如何隔离噪声

SubAgent 的做法是把一个复杂子任务放进独立上下文。

主 Agent 只下发任务目标。SubAgent 自己读取资料、试错、调用工具、整理结果。完成后，它只把摘要或结论返回给主 Agent。

这样，主 Agent 不需要承接全部中间过程。它只保留任务分配和结果整合所需的信息。

## 5. 分工背后是语义边界

如果只把 SubAgent 理解为“更多人手”，就容易盲目拆分。

更好的拆分标准是语义边界：这个子任务是否有独立目标、独立工具、独立上下文、独立验收结果？

如果答案是肯定的，它适合交给 SubAgent。如果只是为了显得架构复杂，就没有必要。

## 6. 一个代码分析例子

让 Agent 分析一个大型代码仓库时，主 Agent 可以负责总目标：理解架构并写报告。

它可以把“阅读测试体系”“分析 API 入口”“检查数据模型”分别交给 SubAgent。

每个 SubAgent 只返回关键发现、证据文件和风险判断。主 Agent 用这些结果合成报告，而不是背负所有阅读过程。

## 7. 这一篇的结论

多 Agent 的表面价值是分工，深层价值是上下文隔离。

SubAgent 真正解决的是：让复杂任务的中间噪声留在局部，把主 Agent 的上下文留给目标、判断和整合。
